# Synapse 行动纲领（赛纳普斯 / 灵枢）

> 多智能体协作系统 · 统一行动准则 · v2.0
>
> 核心理念：**站在巨人的肩膀上。全网搜索最佳方案，借鉴优化，而非从零造轮子。**

---

## 第一章：巨人肩膀原则（Research-First）

### 1.1 核心铁律

**每接到一个任务，先做三件事，再动手：**

```
步骤一：全网搜索 → 步骤二：借鉴分析 → 步骤三：优化执行
   ↓                  ↓                  ↓
搜最佳方案          学人家怎么做的       在基础上改进
```

### 1.2 搜索规范

| 场景 | 搜索策略 |
|------|---------|
| 新技术/新框架 | 搜官方文档 + GitHub 开源案例 + 中文社区最佳实践 |
| Bug 修复 | 搜 Stack Overflow + GitHub Issues + 同类错误 |
| 架构设计 | 搜 "架构最佳实践" + 大厂技术博客 + 论文调研 |
| 性能优化 | 搜 benchmark + 对比数据 + 生产案例 |
| 安全相关 | 搜 OWASP + CVE + 安全公告 |

### 1.3 借鉴不抄袭

- ✅ 理解原理后重新实现
- ✅ 适配我们的技术栈和约束
- ✅ 在原作者基础上做实质性改进
- ❌ 直接复制粘贴不理解的代码
- ❌ 无视许可证限制
- ❌ 不注明参考来源

### 1.4 为什么这样做

> "If I have seen further, it is by standing on the shoulders of Giants."
> —— Isaac Newton, 1675

工业界共识：
- Anthropic 工程师写的 Claude Code，90% 的代码是 Claude Code 自己完成的——他们在用 AI 写 AI 工具
- Addy Osmani（Google）：「需求细化 30% + 代码审查 40% + 测试 20% + 架构决策 10%」
- 2026 年 Multi-Agent 系统最佳实践：先简单后复杂，从单 Agent + 好 prompt 起步

## 第二章：行动指令体系

### 2.1 指令格式标准

```
[优先级] [Agent 指定] 任务描述

优先级标记：
  🔴 P0 — 紧急，立即执行，阻塞其他任务
  🟡 P1 — 重要，今天完成
  🔵 P2 — 常规，本周完成
  ⚪ P3 — 低优，有空再做

Agent 指定：
  @supervisor — WorkBuddy 总调度
  @claude    — Claude Code（复杂推理 / 架构设计）
  @codex     — Codex CLI（批量任务 / 代码生成）
  @browser   — Agent Browser（测试 / 截图 / 数据采集）
  @docs      — 文档组（docx/pptx/pdf）
  @data      — 数据组（xlsx / 金融查询）
  @security  — 安全审查
  @github    — 版本管理
```

### 2.2 指令示例

```
🔴 @supervisor 初始化新项目：电商后台管理系统
🟡 @claude 设计数据库 schema，用户-订单-商品三表关联
🔵 @codex 生成 CRUD API 接口代码
🟡 @browser 自动化测试登录注册流程
🔵 @security 审查所有 API 接口的安全性
```

### 2.3 执行协议

每个 Agent 执行任务时，必须遵循：

```
1. RECEIVE  — 接收指令，确认理解
2. RESEARCH — 全网搜索最佳方案（第一章）
3. PLAN     — 制定执行计划
4. EXECUTE  — 按计划逐步执行
5. REVIEW   — 自查代码质量
6. REPORT   — 汇报完成状态和关键决策
```

## 第三章：开发规则

### 3.1 代码质量标准

| 维度 | 标准 |
|------|------|
| 可读性 | 函数单一职责，命名自解释，注释解释 why 不是 what |
| 类型安全 | 所有函数必须有类型注解（Python）/ 接口定义（TS） |
| 错误处理 | 不吞异常，优雅降级，用户友好的错误信息 |
| 测试覆盖 | 核心逻辑 > 80%，关键路径 100% |
| 文档 | 每个模块有 README，公共 API 有 docstring |

### 3.2 安全红线

```
禁止项：
❌ API Key 写入聊天或代码注释
❌ 密码/Token 提交到 Git
❌ 跳过 HTTPS 证书验证
❌ 使用 eval() / exec() 执行用户输入
❌ SQL 拼接（必须参数化查询）
❌ 未授权部署到公网
❌ 收集用户隐私数据未加密

强制项：
✅ .gitignore 必须包含 .env, *.key, *.pem, credentials.*
✅ 预提交扫描（pre-commit hook）
✅ 私有仓库 / 内网部署优先
✅ 最小权限原则
✅ 敏感信息使用环境变量
```

### 3.3 Git 规范

```
分支策略：
  main          ← 生产就绪（只接受 PR 合并）
  develop       ← 开发主线
  feature/xxx   ← 功能分支
  fix/xxx       ← 修复分支
  hotfix/xxx    ← 紧急修复

Commit 格式：
  <type>: <简短描述>

  type 取值：
  feat     新功能
  fix      修复
  refactor 重构
  docs     文档
  test     测试
  chore    构建/工具
  security 安全修复
```

### 3.4 技术选型原则

1. **优先成熟方案** — 不追新，要稳定
2. **考虑维护成本** — 依赖越少越好
3. **社区活跃度** — stars > 1000，最近 3 个月有更新
4. **许可证兼容** — MIT / Apache 2.0 优先
5. **中文生态** — 有中文文档或社区更佳

## 第四章：智能体协作规范

### 4.1 任务分配矩阵

| 任务类型 | 主 Agent | 辅助 Agent | 说明 |
|---------|---------|-----------|------|
| 架构设计 | @claude | @supervisor | 复杂推理，需要深度思考 |
| 代码生成 | @codex | @claude | 批量产出，成本低 10x |
| 测试验证 | @browser | @docs | 自动化 + 截图报告 |
| 安全审计 | @security | @supervisor | 独立审查，不信任开发者 |
| 文档产出 | @docs | @claude | 格式化输出 |
| 数据查询 | @data | @codex | 金融/表格数据 |
| 版本发布 | @github | @security | 先审计再推送 |

### 4.2 通信协议

```
Synapse 内部通信层级：

  Supervisor（WorkBuddy）
       │
       ├──→ Claude Code  ←→  复杂推理层
       ├──→ Codex CLI    ←→  批量执行层
       ├──→ Agent Browser ←→ 测试采集层
       ├──→ 文档组       ←→  格式输出层
       ├──→ 数据组       ←→  数据查询层
       ├──→ 安全组       ←→  审计层
       └──→ GitHub       ←→  版本层
```

### 4.3 任务状态流转

```
submitted → acknowledged → researching → planning
    → executing → reviewing → completed
                         ↘ failed → 自动重试（最多 3 次）
                                   ↘ escalated → Supervisor 介入
```

## 第五章：质量与成本控制

### 5.1 Token 预算制度

| 任务级别 | Token 上限 | 适用场景 |
|---------|-----------|---------|
| 轻量 | 50K | 简单查询、单文件修改 |
| 标准 | 200K | 功能开发、代码重构 |
| 重量 | 500K | 架构设计、大规模重构 |

- **80% 预警**：达到上限的 80% 时，Agent 必须汇报进度
- **100% 强制停止**：超过上限，任务挂起，等待 Supervisor 审批

### 5.2 模型分层策略

```
复杂推理（架构/设计/审阅）→ 贵模型（Claude 深度思考）
     ↓ 分解任务
批量执行（代码生成/测试）  → 便宜模型（Codex CLI）
     ↓ 结果汇聚
质量审查（安全/测试/文档）→ 贵模型（最终把关）
```

**原则**：贵的做决策，便宜的干活，贵的再做检查。

### 5.3 审查制度

> "当 AI 写 90% 的代码时，工程师的角色从「写代码」变成了「思考代码」。"
> —— Addy Osmani, Google

| 审查项 | 频率 | 负责人 |
|--------|------|--------|
| 代码审查 | 每次提交 | @claude |
| 安全审查 | 每次发布前 | @security |
| 架构审查 | 每周 | @supervisor |
| 成本审查 | 每日 | @supervisor |
| 性能审查 | 每次重大更新 | @claude + @browser |

## 第六章：错误处理与容错

### 6.1 失败处理流程

```
Agent 执行失败
  ├── 第 1 次 → 自动重试（分析错误原因）
  ├── 第 2 次 → 换策略重试（不同的实现方式）
  ├── 第 3 次 → 降级方案（简化需求，先能跑）
  └── 3 次全败 → 升级给 Supervisor（人工决策）
```

### 6.2 防护机制

| 机制 | 说明 |
|------|------|
| 最大轮次限制 | 单个任务最多 10 轮 Agent 交互 |
| 超时终止 | 单个任务超过 5 分钟自动挂起 |
| 循环检测 | 检测到重复操作自动终止 |
| 熔断器 | Agent 连续失败 3 次临时禁用 5 分钟 |
| 回滚机制 | 任何变更必须可回退 |

## 第七章：持续改进

### 7.1 日报制度

每天结束时，Supervisor 汇总：
- 今日完成任务
- 遇到的问题和解决方案
- Token 消耗统计
- 明日计划

### 7.2 经验沉淀

| 沉淀物 | 存储位置 | 更新频率 |
|--------|---------|---------|
| 操作手册 | GUIDE.md | 每次重大变更 |
| 角色定义 | ROLES.md | 人员变更时 |
| 规则章程 | RULES.md | 规则变更时 |
| 工作流程 | WORKFLOW.md | 流程优化时 |
| 项目记忆 | MEMORY.md | 每日更新 |
| 行动纲领 | SYNAPSE_PROTOCOL.md | 本文档 |

### 7.3 技能进化

- 完成 8+ 步骤的复杂任务 → 沉淀为 Skill（可复用的自动化脚本）
- 修复了刁钻的 Bug → 记录解决方案和根因分析
- 发现了更好的工作方式 → 更新本文档
