## 第一章：安全铁律（全员必读）

### 1.1 密钥管理

| 规则 | 责任人 | 违规级别 |
|------|--------|----------|
| API Key 绝不入聊天记录 | WorkBuddy | 🔴 一级 |
| Key 通过本地文件传递（不在聊天暴露） | 全员 | 🔴 一级 |
| 聊天中暴露的 Key 必须 5 分钟内轮换 | WorkBuddy | 🔴 一级 |
| Key 不入 Git 仓库（.gitignore 强制排除） | O1版本管理 | 🔴 一级 |
| 所有 Key 使用环境变量或专用配置文件存储 | T3开发 | 🟡 二级 |
| 生产 Key 与测试 Key 分离，不混用 | WorkBuddy | 🟡 二级 |

### 1.2 代码安全

| 规则 | 责任人 | 触发时机 |
|------|--------|----------|
| 每次提交前检查是否包含密钥/密码 | O2安全审查 | 每次 commit |
| 新依赖安装前审查许可证和已知漏洞 | O2安全审查 | 每次 npm install |
| 禁止在代码注释中写密码、Token、手机号 | T3/T4开发 | 代码编写时 |
| 第三方 API 调用使用 HTTPS，禁用 HTTP | T3/T4开发 | 接口设计时 |
| 敏感配置文件设置 600 权限 | O1版本管理 | 文件创建后 |
| 外部数据输入必须做校验和转义 | T3开发 | 数据处理时 |

### 1.3 数据隐私

| 规则 | 责任人 |
|------|--------|
| 未经授权不收集、不存储、不上传用户数据 | 全员 |
| 本地代码产物默认为 Private，不推公网 | 全员 |
| 用户个人信息不出现在日志、截图、文档中 | Q1/Q2质量组 |
| 公网部署前必须获得 WorkBuddy 显式批准 | O1版本管理 |
| 第三方服务接入前评估隐私合规风险 | O2安全审查 |

## 第二章：上传与传输安全

### 2.1 代码上传（GitHub）

```
提交前检查清单（由 O2 安全审查员执行）:
□ 1. 扫描提交中是否包含 API Key/Token/密码
□ 2. 确认 .gitignore 包含 *.key, *.pem, .env, credentials.*
□ 3. 检查是否有超过 1MB 的二进制文件（应使用 Git LFS）
□ 4. 确认仓库可见性为 Private
□ 5. 审查 commit message 不含敏感信息
```

### 2.2 文件加密方案

| 场景 | 方案 | 工具 |
|------|------|------|
| 配置文件传输 | 本地文件直读，不入聊天 | 文件系统 |
| 敏感源码同步 | GPG 签名 + 加密压缩 | gpg / 7z AES-256 |
| 数据库备份 | AES-256 加密后存储 | openssl enc |
| 环境变量传递 | 写入 .env → 读入内存 → 删除文件 | Node.js |

**推荐加密命令模板：**
```bash
# 加密: AES-256
openssl enc -aes-256-cbc -pbkdf2 -in secret.txt -out secret.enc

# 解密: 
openssl enc -aes-256-cbc -pbkdf2 -d -in secret.enc -out secret.txt

# 压缩+加密
7z a -p -mhe=on encrypted.7z /path/to/files
```

### 2.3 Git 安全配置

```bash
# 全局 .gitignore 必须包含
*.key
*.pem  
*.p12
.env
.env.*
credentials.*
secret.*
*.enc
*.pwd
```

## 第三章：自动化安全防线

### 3.1 三道防线

```
第一道: 开发时 → .gitignore + .env 隔离 + 代码审查
第二道: 提交前 → 密钥扫描（手工检查）+ .gitignore 确认
第三道: 推送后 → GitHub Private 仓库 + 分支保护
```

### 3.2 安全审查清单（O2 执行）

| 检查项 | 频率 | 方法 |
|--------|------|------|
| Key 泄露扫描 | 每次提交 | 手工检查 + grep 扫描 |
| 依赖漏洞 | 每周 | npm audit / snyk test |
| 仓库权限 | 每月 | GitHub Settings 审查 |
| License 合规 | 引入新依赖时 | 查看 package.json license 字段 |
| 端口暴露 | 部署前 | netstat -ano 检查 |

## 第四章：各角色安全职责

### D1 WorkBuddy（Synapse 总调度）— 安全总负责
- 监督所有安全规则的执行
- 决定安全问题的优先级和处理方式
- 安全事故的第一响应人
- 定期审查团队安全状态

### O2 安全审查员 — 安全执行
- 执行每次提交前的密钥扫描
- 维护 .gitignore 规则
- 审查第三方依赖
- 出具安全审查报告
- **当前状态: 手工执行，由 WorkBuddy 代管**

### O1 GitHub CLI — 版本安全
- 确保仓库为 Private
- 配置分支保护规则
- 管理协作者权限
- 审查 PR 中是否包含敏感文件

### T1-T4 开发组 — 代码安全
- 使用参数化查询（防 SQL 注入）
- 输入校验和转义（防 XSS）
- HTTPS 仅用于外部通信
- 不在代码中硬编码任何凭证

### Q1 Agent Browser — 测试安全
- 测试截图不包含敏感数据
- 测试用数据使用假数据，不用真实用户数据

### Q2 文档组 — 文档安全
- 产品文档不包含内部 IP、端口、密钥
- 对外文档需经安全审查后发布

## 第六章：开发规范细则

### 6.1 文件命名规范
- 项目文件: `项目名-功能.扩展名`
- 文档: `YYYY-MM-DD-标题.md`
- 禁止: 空格、中文特殊符号、超过 50 字符

### 6.2 代码风格
- 缩进: 2 空格
- 引号: 单引号（JS/TS）, 双引号（HTML 属性）
- 分号: 必须
- 行宽: 不超过 120 字符

### 6.3 分支策略
- master/main: 稳定版本，禁止直接推送
- dev: 开发分支
- feat/xxx: 功能分支
- fix/xxx: 修复分支
- 合并前必须经过审查

### 6.4 环境隔离
- dev: 本地开发环境
- staging: 测试环境（可选）
- prod: 生产环境（需授权）

## 第七章：角色激活流程

```
1. 任务触发 → WorkBuddy 评估需要哪些角色
2. 激活角色 → 每个角色按本章程规定的职责执行
3. 安全检查 → O2（或 WorkBuddy 代管）在关键节点执行审查
4. 交付汇报 → WorkBuddy 汇总结果 + 安全检查结论
5. 角色释放 → 任务完成后解除激活
```
